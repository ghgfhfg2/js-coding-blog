섬 `0`부터 `n - 1`까지와 양방향 다리 목록 `bridges`가 주어집니다. 이후 `removals` 배열에 적힌 순서대로 다리가 하나씩 무너집니다.

각 다리가 무너진 **직후마다**, 현재 남아 있는 다리들만 사용했을 때 만들 수 있는 **가장 큰 연결된 섬 무리의 크기**를 구해 배열로 반환하는 `solution` 함수를 작성하세요.

여기서 연결된 섬 무리의 크기는 서로 도달 가능한 섬의 개수입니다.

## 제한사항
- `1 <= n <= 100000`
- `0 <= bridges.length <= 200000`
- `1 <= removals.length <= bridges.length`
- `bridges[i] = [a, b]`
- `0 <= a, b < n`
- `a !== b`
- 같은 두 섬을 잇는 다리는 중복해서 주어지지 않습니다.
- `removals`의 각 값은 `bridges`의 유효한 인덱스이며, 서로 중복되지 않습니다.
- 반환값은 `removals.length`와 같은 길이의 배열입니다.

## 예시
- 입력:
  ```js
  solution(5, [[0, 1], [1, 2], [3, 4], [2, 3]], [3, 1])
  ```
  출력: `[3, 2]`
  - 처음에는 모든 섬이 하나로 이어져 있습니다.
  - `bridges[3] = [2, 3]`이 무너지면 `{0,1,2}`와 `{3,4}`로 나뉘어 가장 큰 크기는 `3`입니다.
  - 그다음 `bridges[1] = [1, 2]`까지 무너지면 `{0,1}`, `{2}`, `{3,4}`가 되어 가장 큰 크기는 `2`입니다.

- 입력:
  ```js
  solution(4, [[0, 1], [1, 2], [2, 3]], [0, 1, 2])
  ```
  출력: `[3, 2, 1]`

- 입력:
  ```js
  solution(6, [[0, 1], [1, 2], [2, 0], [2, 3], [3, 4], [4, 5]], [3, 4])
  ```
  출력: `[3, 3]`

## 힌트
- 다리를 **없애는 방향**으로 생각하면 현재 연결 상태를 빠르게 갱신하기가 까다롭습니다.
- 반대로, 마지막 상태에서 시작해 다리를 **다시 추가하는 방향**으로 뒤집어 보면 어떤 일이 쉬워지는지 생각해 보세요.
- 연결 여부와 컴포넌트 크기를 자주 합쳐야 하므로, 분리 집합(Union-Find / DSU)이 잘 맞습니다.

## 해설
정방향으로 보면 문제는 이렇습니다.

1. 다리 하나가 무너진다.
2. 연결 구조가 바뀐다.
3. 가장 큰 연결 컴포넌트 크기를 다시 알아내야 한다.

하지만 Union-Find는 **간선 추가**에는 강하고, **간선 삭제**에는 직접 대응하기 어렵습니다.
그래서 이 문제는 시간을 뒤집는 것이 핵심입니다.

### 1) 먼저 "모든 제거가 끝난 뒤" 상태를 만든다
`removals`에 포함된 다리들은 결국 다 사라집니다.
따라서 처음부터 이 다리들을 제외하고, **끝까지 살아남는 다리들만** Union-Find에 합칩니다.

이렇게 하면 우리는 "모든 붕괴가 끝난 마지막 그래프"를 얻게 됩니다.

### 2) 제거를 역순으로 되돌린다
이제 `removals`를 뒤에서부터 보면서, 방금 사라졌던 다리를 하나씩 **복구**합니다.

예를 들어 정방향 제거 순서가:
- 3번 다리 제거
- 1번 다리 제거

라면 역순에서는:
- 1번 다리 복구
- 3번 다리 복구

가 됩니다.

역순에서 다리를 하나 복구하기 **직전의 상태**는, 정방향에서 해당 다리까지 제거한 **직후의 상태**와 정확히 같습니다.
즉 역순을 따라가며 매번 현재 최대 컴포넌트 크기를 기록해 두면, 정방향 정답을 복원할 수 있습니다.

### 3) Union-Find에 컴포넌트 크기를 함께 저장한다
각 루트마다 현재 컴포넌트 크기를 저장해 두면, `union(a, b)`가 성공할 때 새 루트의 크기를 갱신할 수 있습니다.
그리고 그 값을 이용해 전체 최대 크기 `maxSize`도 갱신합니다.

핵심 흐름은 다음과 같습니다.

1. `removals`에 포함된 간선을 표시한다.
2. 표시되지 않은 간선만 먼저 모두 Union-Find에 합친다.
3. `removals`를 역순 순회한다.
4. 각 단계에서 현재 `maxSize`를 답 배열에 기록한다.
5. 그다음 해당 간선을 복구하며 `union`한다.
6. 마지막에 기록한 답 배열을 뒤집으면 정방향 답이 된다.

### 왜 이 풀이가 맞는가?
정방향에서 `k`번째 제거가 끝난 상태는,
역순에서 마지막 그래프부터 시작해 `k`번째 제거 이후에 사라졌던 다리들만 다시 복구하지 않은 상태와 같습니다.

즉 두 상태의 활성 간선 집합이 동일합니다.
따라서 역순에서 기록한 컴포넌트 최대 크기를 뒤집으면, 문제에서 요구한 각 제거 직후의 정답과 일치합니다.

### 시간 복잡도
- 초기 간선 합치기: `O(m α(n))`
- 역순 복구: `O(q α(n))`
- 전체: `O((m + q) α(n))`
  - `m = bridges.length`
  - `q = removals.length`

여기서 `α(n)`은 매우 느리게 증가하는 아커만 역함수라서, 사실상 거의 상수처럼 동작합니다.

이 문제의 포인트는 그래프 자체보다, **삭제 문제를 역순 추가 문제로 뒤집어 DSU에 맞는 형태로 바꾸는 발상**입니다.
