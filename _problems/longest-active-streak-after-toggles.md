`1`번부터 `n`번까지 일렬로 놓인 서버가 있습니다. 각 서버는 **가동 중(1)** 이거나 **고장(0)** 상태입니다.

처음에는 모든 서버가 가동 중이며, `brokenIndices`에 들어 있는 위치만 고장 상태입니다. 이후 `toggles`에 적힌 위치를 순서대로 처리하면서 해당 서버의 상태를 **가동 ↔ 고장** 으로 뒤집습니다.

각 토글 직후마다, 현재 상태에서 만들 수 있는 **가장 긴 연속 가동 구간의 길이**를 구해 배열로 반환하는 `solution` 함수를 작성하세요.

## 제한사항
- `1 <= n <= 200000`
- `0 <= brokenIndices.length <= n`
- `1 <= toggles.length <= 200000`
- `brokenIndices`의 원소는 서로 다릅니다.
- `toggles`의 각 원소는 `1` 이상 `n` 이하의 정수입니다.
- 같은 위치가 여러 번 토글될 수 있습니다.
- 각 토글 직후의 최장 연속 가동 구간 길이를 순서대로 반환해야 합니다.

## 예시
- 입력: `n = 8`, `brokenIndices = [2, 6]`, `toggles = [6, 3, 2, 7]` → 출력: `[6, 5, 5, 3]`
  - 초기 상태: `1 0 1 1 1 0 1 1`
  - `6` 토글 후: `1 0 1 1 1 1 1 1` → 최장 `6`
  - `3` 토글 후: `1 0 0 1 1 1 1 1` → 최장 `5`
  - `2` 토글 후: `1 1 0 1 1 1 1 1` → 최장 `5`
  - `7` 토글 후: `1 1 0 1 1 1 0 1` → 최장 `3`
- 입력: `n = 1`, `brokenIndices = [1]`, `toggles = [1, 1, 1]` → 출력: `[1, 0, 1]`
- 입력: `n = 5`, `brokenIndices = []`, `toggles = [3, 3, 1, 5]` → 출력: `[2, 5, 4, 3]`

## 힌트
- 매 토글마다 배열 전체를 다시 훑으면 `O(n * q)`가 되어 너무 느립니다.
- 어떤 구간의 답을 알기 위해서는 그 구간의
  - 왼쪽부터 이어지는 연속 가동 길이
  - 오른쪽부터 이어지는 연속 가동 길이
  - 내부 최장 연속 가동 길이
  를 알고 있으면 병합할 수 있습니다.
- 한 점만 바뀌는 갱신이므로, 세그먼트 트리 같은 자료구조를 떠올려 보세요.

## 해설
이 문제의 핵심은 **한 칸만 바뀌어도 전체 최장 연속 구간이 멀리 떨어진 곳까지 영향을 받을 수 있다**는 점입니다.

예를 들어 가운데 한 칸이 고장에서 가동으로 바뀌면, 왼쪽 연속 구간과 오른쪽 연속 구간이 한 번에 이어져 훨씬 긴 구간이 될 수 있습니다. 그래서 단순히 주변 몇 칸만 보고는 답을 갱신할 수 없습니다.

### 1) 각 구간에서 무엇을 저장해야 하나?
세그먼트 트리의 각 노드가 어떤 구간 `[l, r]`를 담당한다고 합시다.

이때 아래 정보를 저장하면 두 자식 구간을 합쳐 부모 구간 정보를 만들 수 있습니다.

- `len`: 구간 길이
- `prefix`: 구간의 왼쪽 끝에서 시작하는 연속 가동(1) 길이
- `suffix`: 구간의 오른쪽 끝에서 끝나는 연속 가동(1) 길이
- `best`: 구간 내부에서 만들 수 있는 최장 연속 가동 길이

리프 노드라면 값이 간단합니다.
- 해당 위치가 가동 중이면 `prefix = suffix = best = 1`
- 고장 상태면 `prefix = suffix = best = 0`

### 2) 두 자식 노드를 어떻게 병합하나?
왼쪽 자식 `L`, 오른쪽 자식 `R`이 있을 때 부모 노드는 이렇게 계산할 수 있습니다.

- `prefix`
  - 기본값은 `L.prefix`
  - 만약 왼쪽 구간 전체가 모두 1이라면 `L.len + R.prefix`
- `suffix`
  - 기본값은 `R.suffix`
  - 만약 오른쪽 구간 전체가 모두 1이라면 `R.len + L.suffix`
- `best`
  - `max(L.best, R.best, L.suffix + R.prefix)`

마지막 항 `L.suffix + R.prefix`가 중요합니다.
최장 구간이 왼쪽 구간 끝과 오른쪽 구간 시작을 **가로질러 이어질 수 있기 때문**입니다.

### 3) 왜 점 토글에 잘 맞나?
토글은 한 위치의 값만 `0 ↔ 1`로 바꿉니다.

그러면 세그먼트 트리에서 해당 리프만 갱신하고, 루트까지 올라오며 병합 정보를 다시 계산하면 됩니다.

- 한 번의 토글 갱신: `O(log n)`
- 매번 전체 답 확인: 루트의 `best` 값 하나면 충분

따라서 전체 시간 복잡도는 `O((n + q) log n)` 수준으로 처리할 수 있습니다.

### 4) 손으로 검증할 때 자주 하는 실수
이 문제는 토글이 누적되므로, 이전 상태를 정확히 기억하지 않으면 금방 꼬입니다.
특히 같은 위치가 여러 번 등장하면
- 처음 토글에서는 `0 → 1`
- 다음 토글에서는 다시 `1 → 0`
가 되므로, 단순히 “고친다” 혹은 “망가뜨린다”로 고정해서 생각하면 틀립니다.

### 5) 예시 흐름 하나 보기
`n = 5`, 초기 상태가 모두 `1`이고 `toggles = [3, 3, 1, 5]` 라면

- 초기: `1 1 1 1 1`
- `3` 토글 후: `1 1 0 1 1` → 최장 `2`
- `3` 토글 후: `1 1 1 1 1` → 최장 `5`
- `1` 토글 후: `0 1 1 1 1` → 최장 `4`
- `5` 토글 후: `0 1 1 1 0` → 최장 `3`

그래서 정답은 `[2, 5, 4, 3]` 입니다.

이 문제의 포인트는 단순한 합이나 최대값 세그먼트 트리가 아니라, **연속 구간을 병합하기 위한 prefix / suffix / best 정보를 설계하는 것**입니다. 이 병합 규칙을 떠올릴 수 있으면 난도가 확 내려갑니다.
