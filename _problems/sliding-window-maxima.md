정수 배열에서 길이가 `k`인 모든 연속 구간의 최댓값을 차례대로 구하는 `solution` 함수를 작성하세요.

## 문제 설명
정수 배열 `nums`와 정수 `k`가 주어집니다.
배열에서 길이가 정확히 `k`인 **연속 부분 배열(window)** 을 왼쪽부터 한 칸씩 이동시키며,
각 구간의 최댓값을 순서대로 담은 배열을 반환하세요.

예를 들어 `nums = [1, 3, -1, -3, 5, 3, 6, 7]`, `k = 3`이라면 길이 3인 연속 구간은 다음과 같습니다.

- `[1, 3, -1]` → 최댓값 `3`
- `[3, -1, -3]` → 최댓값 `3`
- `[-1, -3, 5]` → 최댓값 `5`
- `[-3, 5, 3]` → 최댓값 `5`
- `[5, 3, 6]` → 최댓값 `6`
- `[3, 6, 7]` → 최댓값 `7`

따라서 반환값은 `[3, 3, 5, 5, 6, 7]`입니다.

## 제한사항
- `1 <= nums.length <= 100000`
- `-1000000000 <= nums[i] <= 1000000000`
- `1 <= k <= nums.length`
- 반환값은 길이 `nums.length - k + 1`인 배열입니다.
- 같은 값이 여러 번 나올 수 있습니다.

## 예시
- 입력: `[1, 3, -1, -3, 5, 3, 6, 7], 3` → 출력: `[3, 3, 5, 5, 6, 7]`
- 입력: `[4, 4, 4, 4], 2` → 출력: `[4, 4, 4]`
- 입력: `[2, 1, 3], 3` → 출력: `[3]`

## 힌트
- 매 구간마다 `k`개를 다시 훑으면 최악의 경우 시간이 너무 오래 걸릴 수 있습니다.
- 현재 구간에서 **최댓값 후보가 될 수 있는 인덱스들만** 남겨 두는 방법을 떠올려 보세요.
- 앞쪽 인덱스가 현재 window 범위를 벗어나면 제거해야 합니다.
- 뒤쪽에 자신보다 작거나 같은 값이 있다면, 더 큰 새 값이 들어왔을 때 그 값들은 앞으로 최댓값이 될 수 없습니다.

## 해설
이 문제의 핵심은 **최댓값 후보만 남기는 단조 덱(monotonic deque)** 입니다.

### 왜 단순 반복이 비효율적일까?
길이가 `k`인 구간이 총 `nums.length - k + 1`개 있습니다.
각 구간마다 다시 `k`개를 모두 확인하면 시간 복잡도는 `O(nk)`가 됩니다.
`n`과 `k`가 모두 커지면 비효율적입니다.

### 어떤 정보를 유지해야 할까?
우리는 현재 window의 최댓값만 빠르게 알고 싶습니다.
이를 위해 **값이 큰 순서대로 유지되는 인덱스 덱**을 사용합니다.

덱에는 실제 값이 아니라 **인덱스**를 저장하는 것이 중요합니다.
그래야 다음 두 가지를 처리할 수 있습니다.

1. 현재 window 범위를 벗어난 원소 제거
2. `nums` 값 비교를 통해 새 후보 정리

### 덱을 어떻게 유지할까?
배열을 왼쪽부터 하나씩 보면서 현재 인덱스를 `i`라고 합시다.

1. **범위를 벗어난 인덱스 제거**
   - window 길이가 `k`이므로, `i - k` 이하의 인덱스는 더 이상 현재 구간에 포함되지 않습니다.
   - 덱의 맨 앞이 범위를 벗어났다면 제거합니다.

2. **뒤에서부터 작은 값 제거**
   - 새 원소 `nums[i]`가 들어오면,
     덱 뒤쪽에 있는 값들 중 `nums[i]`보다 작거나 같은 값은 앞으로 최댓값이 될 수 없습니다.
   - 왜냐하면 새 원소가 더 뒤에 있으면서 값도 더 크거나 같기 때문입니다.
   - 따라서 그런 인덱스들을 뒤에서부터 제거합니다.

3. **현재 인덱스 추가**
   - 이제 `i`를 덱 뒤에 넣습니다.

4. **window가 완성되면 정답 기록**
   - `i >= k - 1`이 되면 길이 `k`인 첫 구간이 완성됩니다.
   - 이때 덱의 맨 앞 인덱스가 현재 구간의 최댓값 위치이므로,
     `nums[deque[0]]`를 정답에 추가합니다.

### 예시 흐름
`nums = [1, 3, -1, -3, 5, 3, 6, 7]`, `k = 3`

- `1` 처리 → 덱: `[1]`
- `3` 처리 → `1`은 3보다 작으므로 제거, 덱: `[3]`
- `-1` 처리 → 덱: `[3, -1]` → 첫 window 최댓값은 `3`
- `-3` 처리 → 덱: `[3, -1, -3]` → 최댓값은 `3`
- `5` 처리 → 뒤의 `-3`, `-1`, `3` 모두 제거 후 덱: `[5]` → 최댓값은 `5`
- 이후에도 같은 방식으로 진행하면 결과는 `[3, 3, 5, 5, 6, 7]`입니다.

### 왜 이 방식이 맞을까?
덱에는 항상 다음 성질이 유지됩니다.

- 인덱스는 앞에서 뒤로 갈수록 증가합니다.
- 값은 앞에서 뒤로 갈수록 엄격히 감소하거나, 같을 때는 뒤의 더 최신 인덱스만 남습니다.
- 따라서 덱의 맨 앞은 항상 현재 window에서 가장 큰 값을 가진 후보입니다.

이 성질 덕분에 각 원소는 덱에 **한 번 들어가고 최대 한 번 나옵니다.**
그래서 전체 시간 복잡도는 `O(n)`입니다.

### 복잡도
- 시간 복잡도: `O(n)`
- 공간 복잡도: `O(k)` ~ `O(n)`

이 문제를 통해 **sliding window에서 매번 전체를 다시 계산하지 않고, 후보 구조를 유지하는 방식**을 익힐 수 있습니다.
