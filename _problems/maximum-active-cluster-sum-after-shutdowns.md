서버를 하나씩 셧다운할 때마다 남아 있는 연속 활성 구간들 중 합이 가장 큰 값을 구하는 `solution` 함수를 작성하세요.

## 문제 설명
정수 배열 `values`와 정수 배열 `removeOrder`가 주어집니다.

- `values[i]`는 i번째 서버의 처리량입니다.
- 처음에는 모든 서버가 활성 상태입니다.
- `removeOrder`는 서버를 **왼쪽부터가 아니라 지정된 순서대로 하나씩 셧다운**하는 인덱스 배열입니다.
- 서버가 셧다운되면 그 위치는 비활성 상태가 되며, 남아 있는 활성 서버들은 **연속한 구간들**로 나뉩니다.

각 셧다운 직후마다, 남아 있는 활성 서버들로 만들 수 있는 **연속 구간 합의 최댓값**을 구해 배열로 반환하세요.
활성 서버가 하나도 없으면 그 시점의 값은 `0`입니다.

예를 들어 `values = [5, 1, 9, 2]`, `removeOrder = [1, 3, 0, 2]`이면:

1. 인덱스 `1` 제거 후: 활성 구간은 `[5]`, `[9, 2]` → 최댓값 `11`
2. 인덱스 `3` 제거 후: 활성 구간은 `[5]`, `[9]` → 최댓값 `9`
3. 인덱스 `0` 제거 후: 활성 구간은 `[9]` → 최댓값 `9`
4. 인덱스 `2` 제거 후: 활성 서버 없음 → `0`

따라서 반환값은 `[11, 9, 9, 0]`입니다.

## 제한사항
- `1 <= values.length <= 100000`
- `1 <= values[i] <= 1000000000`
- `removeOrder.length === values.length`
- `removeOrder`는 `0`부터 `values.length - 1`까지를 한 번씩만 포함하는 순열입니다.
- 반환 배열의 길이는 `values.length`입니다.
- 정답은 JavaScript의 안전한 정수 범위를 넘지 않는다고 가정합니다.

## 예시
- 입력: `values = [5, 1, 9, 2]`, `removeOrder = [1, 3, 0, 2]` → 출력: `[11, 9, 9, 0]`
- 입력: `values = [3, 2, 7, 4, 1]`, `removeOrder = [2, 1, 4, 0, 3]` → 출력: `[5, 5, 4, 4, 0]`
- 입력: `values = [10]`, `removeOrder = [0]` → 출력: `[0]`

두 번째 예시를 보면:

- `2` 제거 후: `[3, 2]`, `[4, 1]` → 최댓값 `5`
- `1` 제거 후: `[3]`, `[4, 1]` → 최댓값 `5`
- `4` 제거 후: `[3]`, `[4]` → 최댓값 `4`
- `0` 제거 후: `[4]` → 최댓값 `4`
- `3` 제거 후: 없음 → `0`

## 힌트
- 서버를 **삭제하는 방향**으로 보면 하나의 큰 구간이 둘로 갈라질 수 있어서 관리가 까다롭습니다.
- 반대로 모든 서버가 꺼진 상태에서 `removeOrder`를 **거꾸로 보며 다시 켠다**고 생각해 보세요.
- 어떤 서버를 다시 켰을 때 양옆이 이미 켜져 있다면, 그 구간들과 합쳐 하나의 더 큰 구간으로 만들 수 있습니다.
- 구간 합과 대표 노드를 함께 관리하려면 union-find(Disjoint Set Union)를 떠올릴 수 있습니다.

## 해설
이 문제를 정방향으로 보면 매번 서버 하나를 제거할 때마다 기존 연속 구간이 둘로 쪼개질 수 있습니다.
이 변화를 빠르게 추적하는 것은 꽤 까다롭습니다.

핵심은 **삭제보다 추가가 쉽다**는 점입니다.
따라서 셧다운 과정을 그대로 따라가지 말고, **역순으로 되돌리기** 합니다.

### 1. 역순으로 생각하기
정방향에서는:
- 처음에 모든 서버가 켜져 있음
- `removeOrder[0]`, `removeOrder[1]`, ... 순서로 서버를 끔

역방향에서는:
- 처음에 모든 서버가 꺼져 있음
- `removeOrder`를 뒤에서부터 보며 서버를 하나씩 다시 켬

어떤 서버를 다시 켜는 순간, 그 서버는 길이 1짜리 활성 구간이 됩니다.
그리고 왼쪽 또는 오른쪽 이웃이 이미 켜져 있다면 같은 연속 구간이므로 합칠 수 있습니다.

### 2. union-find로 무엇을 관리할까?
각 활성 인덱스에 대해 다음을 관리합니다.

- `parent[i]`: 현재 i가 속한 집합의 대표 노드
- `sizeSum[root]`: 그 집합(연속 활성 구간)의 총합
- `active[i]`: 현재 인덱스가 켜져 있는지 여부

서버 `idx`를 새로 켜면:
1. `active[idx] = true`
2. 자기 자신을 대표로 두고 `sizeSum[idx] = values[idx]`
3. `idx - 1`이 활성 상태면 union
4. `idx + 1`이 활성 상태면 union
5. union이 끝난 대표 구간의 합으로 전역 최대값을 갱신

이렇게 하면 매 단계에서 "현재 존재하는 연속 활성 구간들의 합"을 거의 상수 시간에 합칠 수 있습니다.

### 3. 왜 정답 배열을 역순으로 채우나?
역방향에서 어떤 서버를 다시 켜기 **직전**의 최대 구간 합은,
정방향에서 해당 서버를 제거한 **직후**의 최대 구간 합과 같습니다.

그래서 보통 다음 순서로 처리합니다.

- 현재 최대 구간 합을 `answer[i]`에 기록
- 그다음 `removeOrder[i]`를 역순으로 다시 활성화

혹은 뒤에서부터 채우는 방식으로 구현해도 됩니다.
중요한 건 **정방향 삭제 직후 상태와 역방향 추가 직전 상태가 대응**된다는 점입니다.

### 4. 예시 흐름
`values = [5, 1, 9, 2]`, `removeOrder = [1, 3, 0, 2]`

역순 추가 순서는 `2 → 0 → 3 → 1`입니다.

- 시작: 아무것도 없음, 최대합 `0`
- `2` 활성화 → 구간 `[9]`, 최대합 `9`
- `0` 활성화 → 구간 `[5]`, `[9]`, 최대합 `9`
- `3` 활성화 → `2`와 합쳐 `[9, 2]`, 최대합 `11`
- `1` 활성화 → 전체 `[5, 1, 9, 2]`, 최대합 `17`

이 역순 상태들을 정방향 제거 직후에 맞춰 배치하면 `[11, 9, 9, 0]`이 됩니다.

### 5. 복잡도
- 각 인덱스는 한 번만 활성화됩니다.
- 각 union/find는 경로 압축과 union-by-size를 쓰면 거의 상수 시간처럼 동작합니다.
- 전체 시간 복잡도는 `O(n α(n))`에 가깝고, 실전에서는 사실상 선형입니다.
- 추가 메모리는 `O(n)`입니다.

이 문제의 포인트는 **연속 구간 합 자체보다, 삭제 문제를 역순 추가 문제로 바꾸는 발상**입니다.
그 뒤에는 union-find로 인접 활성 구간을 묶어 최대 합만 추적하면 됩니다.
