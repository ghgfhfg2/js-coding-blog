작업 로그를 순서를 유지한 채 여러 날에 나눠 처리할 때, 하루 최대 처리량의 최솟값을 구하는 `solution` 함수를 작성하세요.

## 문제 설명
정수 배열 `tasks`와 정수 `days`가 주어집니다.
`tasks[i]`는 i번째 로그 묶음을 처리하는 데 필요한 시간을 뜻합니다.

로그 묶음은 **주어진 순서를 바꿀 수 없고**, 각 날에는 **연속한 몇 개의 묶음만** 처리할 수 있습니다.
또한 하나의 로그 묶음을 여러 날에 걸쳐 쪼갤 수는 없습니다.

이때 전체 로그를 **`days`일 이내**에 모두 처리하려고 합니다.
하루에 처리해야 하는 시간의 최댓값이 가능한 한 작아지도록 할 때,
그 최댓값의 **최솟값**을 반환하세요.

## 제한사항
- `1 <= tasks.length <= 100000`
- `1 <= tasks[i] <= 1000000000`
- `1 <= days <= tasks.length`
- 로그 묶음의 순서는 반드시 유지해야 합니다.
- 각 날에는 비어 있지 않은 연속 구간 하나를 처리한다고 가정합니다.

## 예시
- 입력: `[7, 2, 5, 10, 8], 2` → 출력: `18`
- 입력: `[5, 5, 5, 5], 4` → 출력: `5`
- 입력: `[1, 2, 3, 4, 5], 2` → 출력: `9`

첫 번째 예시에서 하루 최대 처리량을 `18`로 잡으면 다음처럼 나눌 수 있습니다.

- 1일차: `[7, 2, 5]` → 합 `14`
- 2일차: `[10, 8]` → 합 `18`

`17` 이하로는 어떤 방식으로 나눠도 2일 안에 끝낼 수 없으므로 정답은 `18`입니다.

## 힌트
- 정답 자체를 바로 만드는 대신, **"하루 최대 처리량을 X로 잡았을 때 가능한가?"** 를 판별해 보세요.
- 가능한 값의 범위는 가장 큰 로그 묶음 시간부터 전체 합까지입니다.
- 어떤 최대 처리량이 가능하다면, 그보다 큰 값도 항상 가능합니다.

## 해설
이 문제는 정답을 직접 구성하기보다 **가능 여부를 기준으로 이분 탐색**하는 것이 핵심입니다.

### 1. 왜 이분 탐색이 가능한가?
하루 최대 처리량을 `limit`이라고 합시다.

- 만약 `limit = 20`으로 모든 로그를 `days`일 안에 처리할 수 있다면,
  `21`, `22`, `30`처럼 더 큰 값으로도 항상 처리할 수 있습니다.
- 반대로 어떤 `limit`으로 불가능하다면, 그보다 더 작은 값도 불가능합니다.

즉, `불가능 ... 불가능 | 가능 ... 가능` 형태로 경계가 생기므로 이분 탐색이 가능합니다.

### 2. 판별 함수는 어떻게 만들까?
주어진 `limit` 안에서 왼쪽부터 로그를 순서대로 담습니다.

- 현재 날에 다음 로그를 더해도 `limit`를 넘지 않으면 같은 날에 넣습니다.
- 넘는다면 다음 날로 넘어가서 다시 시작합니다.
- 필요한 날 수가 `days`를 초과하면 그 `limit`은 불가능합니다.

이 방식이 맞는 이유는 순서를 바꿀 수 없고, 각 날은 연속 구간만 처리해야 하므로
현재 날에 최대한 많이 담는 greedy가 필요한 날 수를 최소화하기 때문입니다.

### 3. 탐색 범위
- 최소값: `tasks` 안의 최댓값
  - 어떤 날이든 가장 큰 로그 묶음 하나는 통째로 처리해야 하므로 이보다 작을 수 없습니다.
- 최대값: `tasks` 전체 합
  - 모든 로그를 하루에 몰아서 처리하면 항상 가능합니다.

### 4. 예시 흐름
`tasks = [7, 2, 5, 10, 8]`, `days = 2`를 보겠습니다.

- 최소 후보 `left = 10`
- 최대 후보 `right = 32`
- 중간값 `21` → 가능
- 더 줄여서 `15` → 불가능
- 다시 `18` → 가능
- `17` → 불가능

따라서 가능한 최소값은 `18`입니다.

이 문제는 **연속 구간 분할 + 최소 가능한 최대값** 형태의 대표적인 파라메트릭 서치 문제입니다.
정답 후보를 이분 탐색하고, 각 후보는 greedy 시뮬레이션으로 빠르게 판별하면 됩니다.
